# To-Do リストアプリ 仕様書

開発ロードマップ・スモールステップは [todo-app-plan.md](todo-app-plan.md) を参照。本書はアプリの機能・データ・画面の仕様を定義する。

## 1. 目的・スコープ

- 個人利用・学習目的のシンプルな to-do リスト管理アプリ。
- まずは最小限の機能（MVP）で動くものを作り、小さいステップで段階的に機能を追加していく。
- 将来的に複数端末での同期やユーザー管理が必要になった場合に拡張できる構成にしておく（ただし最初から作り込みすぎない）。

## 2. 想定ユーザー・利用シーン

- 利用者：自分一人（学習目的も兼ねる）
- 利用シーン：日々のタスク管理、買い物リスト、学習タスクの管理など
- 利用規模：個人利用のため、大規模なアクセスやマルチユーザー対応は考慮しない

## 3. 技術スタック

段階的に育てられる構成として、以下を推奨する。

| 項目 | MVP（フェーズ1〜2） | 将来の拡張（フェーズ5以降・任意） |
|---|---|---|
| フロントエンド | React + TypeScript + Vite | 変更なし |
| スタイリング | CSS Modules または Tailwind CSS | 変更なし |
| データ保存 | ブラウザの localStorage | バックエンドAPI + DB（例: Node.js/Express + SQLite or Firebase） |
| ホスティング | GitHub Pages / Vercel など静的ホスティング | サーバー付きホスティング（Vercel Functions, Renderなど） |
| テスト | Vitest + React Testing Library | 同様に拡張 |

> 理由：サーバー不要で個人利用に十分なシンプルさを保ちつつ、React/TypeScript は学習効果も高く、後からバックエンドを追加しやすい構成のため。

## 4. 機能要件（MVP → 拡張の順）

### MVPで実装する機能
1. タスクの追加（タイトルのみ）
2. タスク一覧の表示
3. タスクの完了・未完了の切り替え
4. タスクの削除
5. ページ再読み込み後もデータが残る（localStorage永続化）

### 拡張フェーズで追加する機能（優先度順）
6. タスクの編集（タイトル変更）
7. 期限日の設定・表示
8. タスクの並び替え（作成日順、期限順など）
9. フィルタリング（すべて／未完了／完了済み）
10. カテゴリ・タグ分け
11. 検索機能
12. ダークモード対応
13. （任意）複数端末同期のためのバックエンド + DB導入
14. （任意）ユーザー認証（ログイン機能）

## 5. データモデル（MVP時点）

```ts
interface Task {
  id: string;          // UUID
  title: string;       // タスク内容
  completed: boolean;  // 完了フラグ
  createdAt: string;   // ISO 8601 形式の作成日時
}
```

拡張フェーズで `dueDate`, `tags`, `order` などのフィールドを追加していく。

## 6. 画面設計（ラフ）

- **メイン画面（1画面構成）**
  - ヘッダー：アプリ名
  - タスク追加フォーム（入力欄 + 追加ボタン）
  - タスク一覧（チェックボックス + タイトル + 削除ボタン）
  - （拡張後）フィルタタブ：すべて／未完了／完了済み

詳細なワイヤーフレームは実装前に別途 Figma または手書きスケッチで作成する（必要であれば別ステップで対応）。
